# 全局客户端 IP 黑名单设计

## 背景

现有 SSRF IP 黑名单限制的是服务器访问的出站目标，不会阻止某个来源 IP 登录或调用 API。项目另有 API Token 级 IP 白名单，但它只约束特定 Token，无法全局禁止某个客户端 IP，也不覆盖网页登录和 Session 接口。

本功能提供管理员可配置的全局客户端 IP 黑名单。规则数量预计为几十条，目标是覆盖网站、登录、管理接口和模型接口，同时保持请求路径上的开销可忽略。

## 目标

- 支持通过单个 IPv4、IPv6 地址或 CIDR 网段禁止客户端访问。
- 黑名单启用后覆盖静态页面、登录注册、Session 接口、管理接口和所有模型/中继接口。
- 配置更新后立即在当前实例生效，无需重启。
- 请求处理期间不查询数据库、不解析规则文本。
- 在反向代理或 CDN 部署中明确、可靠地取得真实客户端 IP。
- 提供防止管理员意外封禁当前来源 IP 的确认机制。
- 保留必要的存活检查接口，避免负载均衡器将正常实例判定为故障。

## 非目标

- 不替代 API Token 的 IP 白名单。
- 不改变 SSRF 出站访问控制。
- 不实现自动封禁、威胁情报订阅、失败次数检测或临时封禁。
- 不为几十条规则引入 Radix Tree、外部缓存或独立数据库表。
- 首个版本不提供按用户、路径、时间段或封禁原因的细粒度规则。

## 方案比较

### 方案 A：全局中间件读取预编译内存快照（采用）

配置保存或同步时将字符串规则解析为不可变的 `netip.Prefix` 列表，并以原子快照发布。每个请求只解析一次客户端 IP 并遍历内存列表。

优点是无数据库访问、实现边界清晰、配置可立即生效，并且几十条规则的线性匹配开销极低。缺点是多实例之间仍受现有配置同步周期约束。

### 方案 B：每次请求读取数据库

实现直观，但会把所有页面和 API 请求都变成数据库依赖，增加延迟和故障面，不采用。

### 方案 C：IP 前缀树

适合数万到数十万条规则，但当前规模只有几十条，会增加实现和维护成本，不采用。后续只有在规则规模和基准数据证明线性扫描成为瓶颈时才考虑。

## 配置模型

在系统设置域增加独立配置，不复用 `fetch_setting`，避免继续混淆客户端来源控制和 SSRF 出站控制。

配置键：

- `client_ip_setting.blacklist_enabled`：是否启用全局客户端 IP 黑名单，默认 `false`。
- `client_ip_setting.blacklist`：JSON 字符串数组，元素为单个 IP 或 CIDR，默认空数组。
- `client_ip_setting.trusted_proxies`：JSON 字符串数组，元素为可信反向代理的 IP 或 CIDR，默认空数组。

单个 IP 在编译时规范化为主机前缀：IPv4 使用 `/32`，IPv6 使用 `/128`。输入会去除首尾空白和重复项；非法 IP、非法 CIDR、IPv4/IPv6 位数不匹配等情况必须拒绝保存，不能静默忽略。

配置继续存储在现有 Options/分层配置体系中，因此不新增数据库表，也不涉及跨数据库迁移。

## 客户端 IP 解析与可信代理

黑名单是否可靠取决于客户端 IP 的来源。不能无条件信任客户端传入的 `X-Forwarded-For` 或类似请求头。

解析规则如下：

1. 直接连接地址来自 `RemoteAddr`。
2. 当直接连接地址不属于 `trusted_proxies` 时，忽略所有转发头，以直接连接地址作为客户端 IP。
3. 当直接连接地址属于可信代理时，按照 Gin 的可信代理链规则解析转发头，得到最靠近外部且不受信任的地址。
4. 不能解析出合法 IP 时采用失败关闭策略，返回 403，并记录警告日志。

默认可信代理列表为空。这一默认值不会相信公网请求伪造的转发头。部署在 Nginx、Traefik、Cloudflare 或负载均衡器后的管理员需要显式填写实际代理网段。界面需提示：错误配置可能导致所有请求显示为代理 IP，进而误封全部用户或使规则无法命中。

首个版本使用独立、只读快照驱动的客户端 IP 解析器，不在请求期间修改 Gin Engine 的全局可信代理配置。全局黑名单中间件和“当前识别 IP”接口使用该解析器。现有 Token IP 白名单及按 IP 限速继续使用 `c.ClientIP()`；统一全项目客户端 IP 语义属于后续独立改动，不与本功能捆绑。

## 请求拦截流程

全局中间件安装在路由和静态资源中间件之前，并尽量位于请求 ID 中间件之后，以便拒绝日志具有请求关联信息。

处理流程：

1. 读取当前不可变配置快照。
2. 功能未启用时直接继续。
3. 检查请求路径是否属于明确的健康检查豁免。
4. 解析可信客户端 IP。
5. 遍历预编译前缀；任一规则命中即终止请求并返回 HTTP 403。
6. 未命中则继续后续中间件、静态资源或业务路由。

黑名单覆盖 `/api`、`/v1`、其他中继路由、登录注册接口、管理接口、Web 页面和静态资源。首个版本仅豁免无需鉴权的运行状态接口 `/api/status`。其他接口（包括 `/api/uptime/status`）不豁免。豁免路径不做用户可配置，避免管理员无意扩大绕过范围。

命中时统一返回状态码 403 和稳定错误码，例如 `client_ip_blocked`。响应不得回显完整黑名单。后端警告日志记录请求 ID、规范化客户端 IP 和请求路径，但不记录认证令牌或请求正文。

## 配置更新与内存快照

运行时快照包含：启用状态、规范化前缀列表和可信代理列表。快照在构建完成后一次性原子替换，请求不会观察到半更新状态。

以下路径必须触发重建：

- 应用启动并从数据库加载 Options 后。
- 管理员通过设置接口更新相关配置后。
- 多实例环境中的后台 Option 同步完成后。

只有完整配置通过校验后才发布新快照。若同步到非法的历史值，保留上一份有效快照、记录错误，并避免因空快照而意外放行或封禁。

多实例部署沿用现有 Option 同步模型：当前实例保存后立即生效，其他实例在下一次配置同步后生效。本功能不新增分布式消息系统。

客户端 IP 黑名单使用专用的 Root 管理更新接口，一次请求提交启用状态、黑名单、可信代理和可选的自我封禁确认标记。后端完成整体校验后通过现有批量 Option 更新能力原子持久化，再发布单份新快照。通用的单项 Option 更新接口不得用于修改这三个键，避免部分更新和绕过自我封禁确认。

## 管理界面

在默认前端的“系统设置 → 安全设置”增加“客户端 IP 黑名单”区块，与 SSRF 设置分开显示。

界面包含：

- 启用开关。
- 黑名单多行输入框，一行一个 IP 或 CIDR。
- 可信代理多行输入框，一行一个 IP 或 CIDR。
- 当前请求识别到的客户端 IP，用于管理员核对代理配置。
- 关于全站拦截、健康检查豁免和可信代理风险的说明。

保存前进行前端格式校验，但后端仍是最终校验边界。若新规则会命中发起保存请求的客户端 IP，后端首次返回需要显式确认的冲突响应；前端展示强警告，管理员再次确认后才能保存。确认只针对本次提交，不能成为永久绕过规则。

所有新增用户可见文本必须使用 i18n，并同步 `en`、`zh`、`fr`、`ja`、`ru`、`vi` 六种 locale。

## 防误封与恢复

- 默认关闭，升级后不改变现有访问行为。
- 启用空黑名单等价于不拦截。
- 保存会命中当前管理员 IP 的配置需要二次确认。
- 健康检查保持可访问。
- 如果管理员确认后封禁自己，只能从未被封禁的来源、直接修改 Options 数据或临时关闭功能后恢复；文档需明确这一点。
- 不提供基于请求头、Cookie、管理员身份或秘密参数的旁路，以免形成永久安全后门。

## 性能设计

请求热路径只执行：一次快照读取、一次 IP 解析和最多几十次 `netip.Prefix.Contains`。不进行数据库、Redis、DNS、文件 IO、JSON 解码、字符串规则解析或锁竞争。

线性复杂度为 `O(n)`，其中 `n` 是黑名单规则数。在几十条规则的预期规模下，开销相对于登录、数据库请求或上游模型调用可以忽略。流式响应只在建立请求时检查一次，不对 SSE 数据块重复检查。

实现后应增加确定性的基准测试或微基准作为开发验证，但不能把不稳定的绝对耗时阈值写入普通测试。正确性测试应关注匹配结果和请求是否在业务处理前被终止。

## 测试策略

后端单元测试覆盖：

- 单个 IPv4/IPv6 地址和 CIDR 的命中、未命中。
- 重复规则、空白规则的规范化。
- 非法规则拒绝且旧快照继续有效。
- 功能关闭和空列表不拦截。
- 黑名单命中返回 403 且下游 Handler 未执行。
- 登录、Session、模型 API 和静态资源均经过全局中间件。
- `/api/status` 健康检查豁免。
- 非可信来源伪造 `X-Forwarded-For` 不生效。
- 可信代理链能解析出预期客户端 IP。
- 保存会封禁当前管理员 IP 时要求二次确认。
- 配置更新后新请求读取新快照。

新增或大幅改写的 Go 测试使用 `testify/require` 和 `testify/assert`。数据库相关测试必须兼容 SQLite、MySQL 和 PostgreSQL；本设计不新增表或方言相关 SQL。

前端测试覆盖表单规范化、非法规则提示、自我封禁确认流程和提交值。实现完成后执行相关 Go 测试，以及默认前端的 typecheck、lint、i18n 同步检查和生产构建。

- 更新部署文档，说明可信代理配置方式和常见 Nginx/CDN 示例。
- 更新项目索引中与客户端 IP 安全、可信代理相关的运行说明。
- 明确 SSRF IP 黑名单、Token IP 白名单和全局客户端 IP 黑名单三者的区别。
- 日志与错误响应应便于排查规则命中，但不能暴露完整安全配置。

## 验收标准

- 管理员可以配置几十条单 IP/CIDR 规则并立即生效。
- 命中来源无法访问网页、登录、管理和模型接口，只能访问固定健康检查接口。
- 未命中来源的现有行为不变。
- 请求热路径无数据库和锁竞争，流式请求只检查一次。
- 伪造转发头不能绕过黑名单；可信代理部署可正确识别真实 IP。
- 配置错误不会替换上一份有效快照。
- 自我封禁必须经过明确二次确认。
